## 遇到的问题和解决
1. 问题：前端接口调用最初分散在页面中，既有直接 `request`，也有临时拼接 URL，导致鉴权头、参数格式和错误处理不统一。  
解决：按业务拆分 `frontend/src/api/auth.js`、`users.js`、`birds.js`、`posts.js`，页面只调用业务函数；底层统一复用 `frontend/utils/request.js`，把 token 注入、超时控制、错误返回集中到一个入口维护。

2. 问题：`/api/v1/birds/recognize` 使用 `multipart/form-data` 上传，与普通 JSON 请求链路不同，联调时容易出现字段名和响应解析不一致。  
解决：在 `birds` 模块独立封装 `wx.uploadFile`，固定上传字段 `image`，并对齐通用响应结构 `code/message/data` 的解析逻辑；同时将识别流程纳入测试覆盖，保证上传链路可回归。


## 心得体会
这次我最明显的收获是对分层和统一约定价值的理解更深了：页面层只负责交互，API 层负责请求组织和协议适配，后端路由层负责参数校验，服务层负责业务逻辑，这样在定位问题时路径更清晰，改动时影响范围也更可控。同时，统一接口前缀 `/api/v1`、统一响应结构 `code/message/data`、统一请求封装后，前后端联调的沟通成本明显下降，很多重复问题都能在规范层面提前避免。

另一个体会是测试对交付质量的作用远比预期更大，把登录、识别、帖子 CRUD、点赞评论这些核心流程固化为可重复执行的测试后，每次改动都能快速确认是否引入回归问题，可靠性比单纯手工联调更高。后续我会继续补充异常场景测试（未登录、参数错误、资源不存在、权限不足），并逐步推动 mock 数据流程向真实数据库流程迁移，让接口行为更贴近真实部署环境。
