### 2. 技术选型
- 前端框架的选择及理由：

本项目前端采用原生微信小程序框架开发。主要原因是项目目标平台就是微信生态，原生方案在页面渲染、API 调用和调试链路上最直接，和微信开发者工具配合也更稳定。相比额外引入跨端框架，原生开发可以减少适配成本，让团队把精力集中在识别流程、页面交互和接口联调上，更适合当前课程项目的开发节奏和实现深度。

- 后端框架的选择及理由：

后端选择 FastAPI。FastAPI 在接口开发效率、参数校验和文档生成方面优势明显，适合当前“快速迭代 + 明确接口规范”的项目场景。它基于 Python 生态，也更便于后续对接智能识别服务，在同一技术栈下完成业务服务与模型服务协作，能降低系统集成复杂度并提升后续维护效率。

## 心得体会
在架构设计过程中，最大的收获是对“合适比先进更重要”这句话有了更深的体会。一开始容易陷入技术选型的纠结，总想用最新最热门的组件，后来发现观鸟社区这种业务场景，数据关系清晰、查询模式固定，用MySQL加Redis的经典组合就完全够用，过度设计反而增加运维成本。另一个收获是明确了读写分离和缓存的介入时机，不是一开始就要把所有优化手段都用上，而是根据业务增长节奏逐步引入，比如先保证索引合理，等查询压力上来了再上从库和缓存。还有就是对数据一致性有了更具体的认识，像点赞防重、评论层级删除这些场景，用数据库自带的外键约束和唯一索引就能优雅解决，比在业务代码里兜一圈要可靠得多。总的来说，这次架构设计让我更清楚如何在功能需求、开发效率和运维成本之间做取舍，而不是单纯追求技术上的“酷”。
